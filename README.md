## Overview

`img2ascii` is a single-file, client-side tool that reads pixel luminance from the HTML5 Canvas API and maps it to a character grid in real time. It ships two rendering modes — ASCII character mapping and Unicode Braille — along with a built-in preset system, a live character map editor, drag-and-drop and clipboard upload, and `.txt` export.

No build step. No package manager. No backend. The entire application lives in one `index.html`.

---

## Features

| Feature | Detail |
|---|---|
| **ASCII rendering** | Maps per-pixel luminance to a sorted character density palette via linear interpolation |
| **Braille rendering** | Groups 2×4 pixel blocks into a single Unicode Braille codepoint (U+2800–U+28FF), producing ~4× effective resolution |
| **Built-in presets** | Classic (` .:-=+*#%@`), Minimal (` :+#`), Blocks (` ░▒▓█`), Braille |
| **Custom character map** | Type any character sequence directly; the field updates the output live |
| **Resolution control** | Width slider from 20 to 400 characters; height is auto-calculated to preserve aspect ratio |
| **Invert toggle** | Flips the brightness-to-character mapping — useful when switching between dark and light backgrounds |
| **Real-time rendering** | Canvas reprocesses on every control change; slider input is debounced at 150 ms |
| **Drag & drop / paste** | Drop image files onto the upload zone or paste from clipboard (`Ctrl+V` / `Cmd+V`) |
| **Light / Dark theme** | Persisted to `localStorage`; respects `prefers-color-scheme` on first visit |
| **Copy to clipboard** | Writes output to clipboard with a fallback using `execCommand` for older browsers |
| **Save as `.txt`** | Downloads the current output via `Blob` + object URL |
| **Fully client-side** | No network requests after initial page load; works offline once cached |

---

## Why This Exists

Most online ASCII converters upload your image to a remote server, require an account, or add noise with ads and friction. `img2ascii` does none of that. The image never leaves your browser — the Canvas API handles all pixel processing locally.

The Braille mode is a specific design choice: each Unicode Braille character encodes a 2×4 dot matrix (8 binary pixels per cell), which gives substantially more spatial information per character than any single-character ASCII substitution. For portrait photos or detailed graphics, this makes a visible difference.

```bash
git clone https://github.com/SoufianoDev/img2ascii.git

```

**Option A — open directly**

```bash

python3 -m http.server 8080

```

Then open `http://localhost:8080`.

## Usage

### Basic workflow

1. Drop an image onto the upload zone, click to browse files, or paste from clipboard (`Ctrl+V` / `Cmd+V`).
2. Select a style preset from the dropdown.
3. Adjust the **Resolution Width** slider (20–400 characters). Height scales automatically.
4. Toggle **Invert Values** if the output looks washed out against your current theme.
5. Use **Copy text** or **Save .txt** to export.

### Style presets

| Preset | Character map | Notes |
|---|---|---|
| Braille | Unicode U+2800–U+28FF | Uses 2×4 pixel blocks; character map field is disabled |
| Classic | ` .:-=+*#%@` | 10-character gradient, good general purpose |
| Minimal | ` :+#` | High contrast, works well for logos and simple graphics |
| Blocks | ` ░▒▓█` | Unicode block elements, pixelated look |
| Custom | _(user-defined)_ | Type any string; leftmost character = lightest pixel |

Selecting any built-in preset writes its character map into the field. Editing the field switches the preset selector to **Custom Pattern** automatically.

### Character map ordering

The character map is read left-to-right, where the leftmost character maps to the lightest pixels and the rightmost to the darkest. The label in the UI shows this as **Light → Dark**.

With **Invert Values** checked, the array is reversed before mapping — not after — so the character map string itself stays unchanged.

```
Light ──────────────────────── Dark
 .:-=+*#%@
^                              ^
lightest pixel        darkest pixel
```

### Keyboard / input shortcuts

| Action | Method |
|---|---|
| Load image | Click drop zone, drag file, or `Ctrl+V` / `Cmd+V` |
| Reset everything | Header **Reset** button |
| Toggle theme | Header sun/moon button |
| Export `.txt` | **Save .txt** button in output toolbar |
| Copy output | **Copy text** button in output toolbar |

## Architecture

### Project structure

```
img2ascii/
└── index.html    # Entire application — markup, styles, and logic in one file
```

All CSS (custom properties + Tailwind utility classes), HTML, and JavaScript are contained in `index.html`. There are no external asset files.

### Rendering pipeline

```
Image source (File / Drop / Clipboard)
          │
          ▼
  FileReader.readAsDataURL()
  → new Image() → img.onload
          │
          ▼
  canvas.drawImage(img, 0, 0, targetW, targetH)
  — offscreen <canvas id="hiddenCanvas"> —
  — willReadFrequently: true on context —
          │
          ▼
  ctx.getImageData() → Uint8ClampedArray (RGBA)
          │
     ┌────┴────┐
     │         │
  ASCII      Braille
  mode       mode
     │         │
     ▼         ▼
```

**ASCII mode** — one pixel per character cell:

```
luminance = 0.2126·R + 0.7152·G + 0.0722·B   (ITU-R BT.709)
charIndex = round((luminance / 255) × (charMap.length − 1))
output   += charMap[charIndex]
```

Height is calculated as `floor(charHeight × 0.55)` to compensate for the non-square aspect ratio of monospace glyphs (characters are taller than wide).

**Braille mode** — eight pixels per character cell:

Each 2×4 pixel block maps to a Braille dot number via the standard Unicode Braille bit layout:

```
Pixel (col, row)  →  Braille dot bit
(0,0) → bit 1 (value   1)    (1,0) → bit 4 (value   8)
(0,1) → bit 2 (value   2)    (1,1) → bit 5 (value  16)
(0,2) → bit 3 (value   4)    (1,2) → bit 6 (value  32)
(0,3) → bit 7 (value  64)    (1,3) → bit 8 (value 128)
```

Each pixel is thresholded at luminance 128. If the dot is "on", its bit value is added to a running `dotValue`. The final character is:

```js
String.fromCharCode(10240 + dotValue)  // 10240 = 0x2800
```

Height scales as `floor(pixelHeight / 4)` (one character per 4-pixel row), and line-height is set to `1` instead of the `0.65` used for ASCII mode.

### Debouncing and rendering queue

```js
function queueProcessing(delay = 50) {
    clearTimeout(processTimeout);
    processTimeout = setTimeout(() => {
        requestAnimationFrame(() => generateASCII());
    }, delay);
}
```

- Preset changes and invert toggle: 50 ms delay (default)
- Width slider `input` event: 150 ms delay
- Custom pattern `input` event: 300 ms delay

The loading overlay is shown immediately on queue and hidden 100 ms after the render completes to avoid flicker on fast machines.

### Theme system

On `<html>` before any render, an inline script checks `localStorage` for a saved preference, then falls back to `prefers-color-scheme`. This prevents a flash of the wrong theme on page load.

```js
// Written to localStorage on toggle:
localStorage.setItem('theme', 'dark' | 'light')
```

Colors are defined as raw RGB triplets in CSS custom properties under `:root` (light) and `.dark` (dark), then consumed via Tailwind's `rgb(var(--color-*) / <alpha-value>)` syntax for opacity support.

## Performance

The main cost is `ctx.getImageData()` — a GPU-to-CPU pixel readback. The canvas context is created with `{ willReadFrequently: true }` to hint to the browser to keep pixel data CPU-accessible.

Practical limits:

- Up to ~300 character width renders in under 100 ms on modern hardware in ASCII mode.
- Braille mode samples fewer cells (each cell is 2×4 pixels) but has more logic per cell; throughput is comparable.
- The slider caps at 400 characters wide. Beyond this, performance degrades noticeably on large source images.
- All processing runs on the main thread. Very large images at maximum resolution will produce a brief freeze. A Web Worker offload is on the roadmap.

## Privacy

All image processing is local. The application:

- makes no network requests after initial page load
- does not send your image anywhere
- stores only the theme preference (`localStorage.getItem('theme')`)
- contains no analytics, telemetry, or third-party tracking scripts
- does not access the clipboard in the background — only on explicit user action (`Ctrl+V` / paste event, or clicking Copy)

---

## Accessibility

- Theme toggle responds to `prefers-color-scheme` on first load.
- All interactive controls (dropdown, slider, checkbox, buttons) are standard HTML elements and are keyboard-accessible without custom handling.
- The output `<div>` uses `font-family: 'Fira Code', 'Courier New', monospace` — a consistent monospace stack for predictable character spacing across platforms.

Known limitation: the ASCII/Braille output block is not meaningfully interpretable by screen readers. It is a visual art format. If sharing output publicly, include a text description of the source image alongside it.

## Browser Compatibility

Requires: Canvas API (`getImageData`), `FileReader`, `DataTransfer`, `Clipboard API`.

| Browser | Minimum version | Notes |
|---|---|---|
| Chrome / Edge | 90+ | Full support |
| Firefox | 88+ | Full support |
| Safari | 14+ | Clipboard paste requires a user gesture; `navigator.clipboard` needs explicit permission |
| Opera | 76+ | Full support |

The clipboard write fallback (`document.execCommand('copy')`) is used when `navigator.clipboard.writeText` is unavailable. Internet Explorer is not supported. Clipboard paste does not work on `file://` origins — serve over HTTP.
