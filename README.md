## Overview
`img2ascii` is a single-file, client-side tool that reads pixel luminance from the HTML5 Canvas API and maps it to a high-fidelity Unicode Braille grid in real time. It ships with an advanced rendering engine featuring Edge Sharpening, Gamma Correction, and Floyd-Steinberg Dithering. It also supports full multilingual I18n (English and Arabic) with automatic RTL/LTR layout management.
No build step. No package manager. No backend. The entire application lives in one `index.html`.

## Why This Exists
Most online converters upload your image to a remote server, require an account, or add noise with ads and friction. `img2ascii` does none of that. The image never leaves your browser — the Canvas API handles all pixel processing locally.
The Braille mode is a specific design choice: each Unicode Braille character encodes a 2×4 dot matrix (8 binary pixels per cell), which gives substantially more spatial information per character than any single-character ASCII substitution. For portrait photos or detailed graphics, this makes a visible difference.

The project has no build toolchain. Clone and open.
```bash
git clone https://github.com/SoufianoDev/img2ascii.git
cd img2ascii
```
**Option A — open directly**
```bash
open index.html
```
Works for most use cases. Note: clipboard paste (`Clipboard API`) is blocked on `file://` origins in some browsers. Use option B if paste doesn't work.
**Option B — local HTTP server**
```bash
# Python
python3 -m http.server 8080
# Node (no install needed)
npx serve .
```
Then open `http://localhost:8080`.
No `.env`, no dependencies, no configuration required.

## Usage
### Basic workflow
1. Drop an image onto the upload zone, click to browse files, or paste from clipboard (`Ctrl+V` / `Cmd+V`).
2. Adjust the **Size (Width)** slider (50–400 characters). Height scales automatically.
3. Fine-tune the details:
   - **Edge Sharpening**: Enhances outlines, crucial for crisp lines.
   - **Threshold (Gamma)**: Controls the contrast curve.
   - **Smooth Dithering**: Spreads dots to simulate gradients and shadows.
4. Toggle **Invert Matrix** if the output looks washed out against your current background.
5. Use **Display Zoom** to scale the font size without altering the resolution.
6. Use **Copy text** or **Save .txt** to export.

## Architecture
### Project structure
```
img2ascii/
└── index.html    # Entire application — markup, styles, fonts, and logic in one file
```
All CSS (Tailwind utility classes + custom UI elements), HTML, and JavaScript are contained in `index.html`.
### Rendering pipeline
```
Image source (File / Drop / Clipboard)
          │
          ▼
  FileReader.readAsDataURL()
  → new Image() → img.onload
          │
          ▼
  Intermediate Canvas (Scaling & Sharpening)
  — applies Custom Sharpen Convolution Matrix —
          │
          ▼
  Final Rendering Canvas (Exact Pixel Grid)
  — ctx.getImageData() → Uint8ClampedArray (RGBA)
          │
          ▼
  Gamma Curve & Threshold Calculation
          │
          ▼
  Floyd-Steinberg Dithering (Optional)
          │
          ▼
  Braille Assembly
  — Grouping 2x4 pixel blocks into Unicode Braille
          │
          ▼
  Output injected to DOM
```
**Braille High-Fidelity Rendering** — eight pixels per character cell:
Each 2×4 pixel block maps to a Braille dot number via the standard Unicode Braille bit layout:
```
Pixel (col, row)  →  Braille dot bit
(0,0) → bit 1 (value   1)    (1,0) → bit 4 (value   8)
(0,1) → bit 2 (value   2)    (1,1) → bit 5 (value  16)
(0,2) → bit 3 (value   4)    (1,2) → bit 6 (value  32)
(0,3) → bit 7 (value  64)    (1,3) → bit 8 (value 128)
```
The rendering engine measures the exact font aspect ratio to ensure pixel-perfect scaling. Each pixel's luminance is calculated, normalized, passed through a gamma curve, and optionally dithered using the Floyd-Steinberg algorithm. If the dot is considered "on" (>128 after processing), its bit value is added to the character.
### Multilingual & UI System
The application uses a lightweight I18n engine (`translations` dictionary) to switch between English and Arabic. When the language changes, it dynamically updates `document.documentElement.dir` to handle Left-to-Right (LTR) and Right-to-Left (RTL) layouts automatically. 
Sliders implement a dynamic CSS variable `--slider-dir` to gracefully invert fill directions based on the active language direction.

## Performance
The main cost is `ctx.getImageData()` — a GPU-to-CPU pixel readback — alongside the new Sharpening Matrix and Dithering algorithms. The canvas context is created with `{ willReadFrequently: true }` to hint to the browser to keep pixel data CPU-accessible.
Practical limits:
- Up to ~300 character width renders in ~200 ms on modern hardware.
- The slider caps at 400 characters wide. Beyond this, performance degrades noticeably on large source images.
- All processing runs on the main thread. Very large images at maximum resolution will produce a brief freeze. A Web Worker offload is on the roadmap.

## Privacy
All image processing is local. The application:
- makes no network requests after initial page load
- does not send your image anywhere
- stores only the theme and language preference in `localStorage`
- contains no analytics, telemetry, or third-party tracking scripts
- does not access the clipboard in the background — only on explicit user action (`Ctrl+V` / paste event, or clicking Copy)
---
## Accessibility
- Theme toggle responds to `prefers-color-scheme` on first load.
- Arabic language support provides full RTL mirroring of the interface.
- All interactive controls are standard HTML elements and are keyboard-accessible.
- The output `<div>` uses `font-family: 'Fira Code', 'Courier New', monospace` — a consistent monospace stack for predictable character spacing across platforms.
Known limitation: the Braille output block is not meaningfully interpretable by screen readers. It is a visual art format. If sharing output publicly, include a text description of the source image alongside it.
